# Patches

This directory contains patches for MinGW, GCC, Binutils, and LLVM to support legacy versions of Windows.

Modern MinGW/LLVM/GCC/MSVC only support Windows Vista+ (2006), and SSE2+ (CPU's made after 2003).
With these patches and compiler flag tuning, we are able to produce toolchains that support
Windows NT 4.0 (1996), 2000 (2000), XP(2001), and CPUs down to the original Pentium (1994).

The LLVM patches were taken and modified from [this repo](https://github.com/mon/llvm-mingw-xp).  
Some of the MinGW/GCC patches were taken and modified from [w64devkit](https://github.com/skeeto/w64devkit/tree/master).  
A notable exception is the mingw-rand_s-win2k.patch, which I made myself for MinGW's rand_s used in std::random.

## List of patches and their purpose

### MinGW

[mingw-gendef-silent.patch](./mingw/mingw-gendef-silent.patch) - Makes `gendef` silent: drops the `Found PE...` progress lines it prints to the console, and the copyright/comment header it writes at the top of your *.def* files.

[mingw-headers.patch](./mingw/mingw-headers.patch) - Lowers MinGW's WinSDK version headers to a sane legacy default (`sdkddkver.h` + `winsdkver.h` default down from Server 2003 to XP, and `windef.h`'s standalone `WINVER` fallback to XP), and adds two definitions missing from mingw-w64: `VS_FF_RELEASE` in `winver.h`, and the `IDirect3D9ExOverlayExtension` interface in `d3d9.h` (placed inside the `!D3D_DISABLE_9EX` guard, since it uses the 9Ex-only `D3DDISPLAYMODEEX`/`D3DDISPLAYROTATION` types).

[mingw-rand_s-win2k.patch](./mingw/mingw-rand_s-win2k.patch) - Fixes MinGW CRT `rand_s` incompatibility with Windows NT 4.0/2000, by using [`CryptGenRandom`](https://en.wikipedia.org/wiki/CryptGenRandom) instead of
                                                   XP+ [`RtlGenRandom`](https://learn.microsoft.com/en-us/windows/win32/api/ntsecapi/nf-ntsecapi-rtlgenrandom) for cryptographically secure random number generation.

[mingw-gettimeofday.patch](./mingw/mingw-gettimeofday.patch) - Fixes MinGW CRT `gettimeofday`/`getntptimeofday` on Windows NT 4.0. The stock code holds a *static* reference to
                                                  [`GetSystemTimeAsFileTime`](https://learn.microsoft.com/en-us/windows/win32/api/sysinfoapi/nf-sysinfoapi-getsystemtimeasfiletime) (Windows 2000+), so any program that pulls in `gettimeofday` fails to load on NT 4.0. This
                                                  resolves it dynamically (via `GetProcAddress`) and falls back to a [`GetSystemTime`](https://learn.microsoft.com/en-us/windows/win32/api/sysinfoapi/nf-sysinfoapi-getsystemtime) + [`SystemTimeToFileTime`](https://learn.microsoft.com/en-us/windows/win32/api/timezoneapi/nf-timezoneapi-systemtimetofiletime)
                                                  emulation (both present on every NT release). Applied only for NT4 targets (`_WIN32_WINNT < 0x0500`).

[sdkddkver.h](./mingw/sdkddkver.h) and [winsdkver.h](./mingw/winsdkver.h) - Custom written replacements for these MSVC's headers, with expanded macros and definitions for old Windows.

### GCC

[gcc-stdcall-align.patch](./gcc/gcc-stdcall-align.patch) - Aligns x86 [__stdcall](https://learn.microsoft.com/en-us/cpp/cpp/stdcall) 4 byte stacks with GCC's 16 byte stack alignment expectations, increasing performance slightly.

[gcc-trap-terminate.patch](./gcc/gcc-trap-terminate.patch) - Replaces `std::terminate`'s __std::abort__ function with a __&#95;&#95;builtin_trap__ trap instruction.

[gcc-avx-misaligned.patch](./gcc/gcc-avx-misaligned.patch) - Treats AVX register spills as always unaligned, removing the need for `-Wa,-muse-unaligned-vector-move` when targeting AVX-capable hardware ([GCC PR54412](https://gcc.gnu.org/bugzilla/show_bug.cgi?id=54412)).

[gcc-tzdb-getdynamic.patch](./gcc/gcc-tzdb-getdynamic.patch) - Fixes C++20 compatability by using [GetTimeZoneInformation](https://learn.microsoft.com/en-us/windows/win32/api/timezoneapi/nf-timezoneapi-gettimezoneinformation) instead of
                                                               Vista+ [GetDynamicTimeZoneInformation](https://learn.microsoft.com/en-us/windows/win32/api/timezoneapi/nf-timezoneapi-getdynamictimezoneinformation).

### Binutils

[binutils-dlltool-zero-ordinals.patch](./binutils-gdb/binutils-dlltool-zero-ordinals.patch) - Prevents randomizing function ordinals in libraries, which is good for reproducible builds and making static libraries more compatible.

[gdb-confirm-off.patch](./binutils-gdb/gdb-confirm-off.patch) - Turns off GDB's `confirm` prompt by default; its notion of "potentially dangerous" operations is far too broad and creates needless friction.

[gdb-alternate-main.patch](./binutils-gdb/gdb-alternate-main.patch) - Allows [GDB](https://sourceware.org/gdb/) to pick up Win32 specific entry point function names like __wWinMain__, __mainCRTStartup__, etc.

### LLVM

[compiler-rt-emutls-pre-vista.patch](./llvm/compiler-rt-emutls-pre-vista.patch) - Replaces Vista+ [InitOnceExecuteOnce](https://learn.microsoft.com/en-us/windows/win32/api/synchapi/nf-synchapi-initonceexecuteonce)
                                                                                with [InterlockedCompareExchange](https://learn.microsoft.com/en-us/windows/win32/api/winnt/nf-winnt-interlockedcompareexchange)
                                                                                for [TLS](https://learn.microsoft.com/en-us/windows/win32/procthread/thread-local-storage).

[libcxx-legacy-filesystem.patch](./llvm/libcxx-legacy-filesystem.patch) - Replaces the Vista+ filesystem APIs `libc++`'s `std::filesystem` relies on
                                                                          ([GetFileInformationByHandleEx](https://learn.microsoft.com/en-us/windows/win32/api/winbase/nf-winbase-getfileinformationbyhandleex),
                                                                          [SetFileInformationByHandle](https://learn.microsoft.com/en-us/windows/win32/api/winbase/nf-winbase-setfileinformationbyhandle),
                                                                          [CreateSymbolicLinkW](https://learn.microsoft.com/en-us/windows/win32/api/winbase/nf-winbase-createsymboliclinkw),
                                                                          [GetFinalPathNameByHandleW](https://learn.microsoft.com/en-us/windows/win32/api/fileapi/nf-fileapi-getfinalpathnamebyhandlew))
                                                                          with NT 4.0/2000/XP equivalents (`GetFileInformationByHandle` + `DeviceIoControl`); symlink creation, `canonical()` and `permissions()` degrade to "not supported" on those targets.

[libcxx-legacy-msvcrt-locale.patch](./llvm/libcxx-legacy-msvcrt-locale.patch) - Routes the per-locale `_l` functions `libc++` uses (`_toupper_l`/`_tolower_l`, the `_iswXXX_l`/`_towXXX_l` ctype/wctype set,
                                                                             `_strcoll_l`/`_strxfrm_l`/`_wcscoll_l`/`_wcsxfrm_l`, and `_mbtowc_l`) through a thread-locale guard around the plain functions. These `_l` symbols only exist in
                                                                             msvcr80+/UCRT - some fail to *compile* (mingw doesn't declare them), but others (e.g. `_mbtowc_l`) link as `msvcrt.dll` imports that the legacy system DLL doesn't export,
                                                                             so a C++ program fails to *load* on pre-Vista with "_mbtowc_l could not be located". After this patch `libc++` imports no `_l` symbols from `msvcrt.dll`. (mingw-w64 supplies `_configthreadlocale`.)

[libcxx-legacy-msvcrt-wcrtomb_s.patch](./llvm/libcxx-legacy-msvcrt-wcrtomb_s.patch) - Shims the bounds-checked [`wcrtomb_s`](https://learn.microsoft.com/en-us/cpp/c-runtime-library/reference/wcrtomb-s) (msvcr80+/UCRT only)
                                                                                   with a local lambda built on the always-available [`wcrtomb`](https://learn.microsoft.com/en-us/cpp/c-runtime-library/reference/wcrtomb-wcrtomb-l), for legacy `msvcrt.dll`.

[libunwind-rwmutex-pre-vista.patch](./llvm/libunwind-rwmutex-pre-vista.patch) - Replaces the Vista+ [SRWLOCK](https://learn.microsoft.com/en-us/windows/win32/sync/slim-reader-writer--srw--locks)
                                                                              `libunwind`'s reader/writer mutex uses with a [CRITICAL_SECTION](https://learn.microsoft.com/en-us/windows/win32/sync/critical-section-objects) fallback (available on all NT versions), keeping `libunwind` dependency-free (no winpthreads).

[libcxx-thread-getsysteminfo.patch](./llvm/libcxx-thread-getsysteminfo.patch) - Makes `std::thread::hardware_concurrency()` fall back from the Windows 7+
                                                                              [GetActiveProcessorCount](https://learn.microsoft.com/en-us/windows/win32/api/winbase/nf-winbase-getactiveprocessorcount) to [GetSystemInfo](https://learn.microsoft.com/en-us/windows/win32/api/sysinfoapi/nf-sysinfoapi-getsysteminfo) on NT 4.0/2000/XP. (Needed on LLVM 22.x; 20.x already used GetSystemInfo.)

[llvm-support-pre-vista.patch](./llvm/llvm-support-pre-vista.patch) - Makes the Windows-hosted LLVM **toolchain binaries** (`clang.exe`, `lld`, `llvm-*`, and the shared `libLLVM`/`libclang-cpp` DLLs) actually LOAD and run on Windows XP/2000. Unlike the other LLVM patches (which fix the *runtime libraries* used by compiled programs), this patches LLVM's own `lib/Support` (`Windows/Path.inc` + `Windows/Threading.inc` + `Windows/Memory.inc` + `Windows/Signals.inc` + `tools/llvm-ar/llvm-ar.cpp` + `lib/Support/raw_socket_stream.cpp`), which statically imports Vista+/Win7+ functions
                                                                       (`Path.inc`: [GetFinalPathNameByHandleW](https://learn.microsoft.com/en-us/windows/win32/api/fileapi/nf-fileapi-getfinalpathnamebyhandlew), [SetFileInformationByHandle](https://learn.microsoft.com/en-us/windows/win32/api/winbase/nf-winbase-setfileinformationbyhandle), [SHGetKnownFolderPath](https://learn.microsoft.com/en-us/windows/win32/api/shlobj_core/nf-shlobj_core-shgetknownfolderpath), [SHCreateItemFromIDList](https://learn.microsoft.com/en-us/windows/win32/api/shobjidl_core/nf-shobjidl_core-shcreateitemfromidlist); `Threading.inc`: the processor-group APIs `GetLogicalProcessorInformationEx` / `Get`/`SetThreadGroupAffinity` / `GetProcessGroupAffinity`; `Memory.inc`: `GetLargePageMinimum`; `Signals.inc`: `RegGetValueW` — its docs claim "minimum client: XP" but it is **not** in stock XP `advapi32.dll`, so it is effectively Vista+ and crashes on XP; `llvm-ar.cpp` (also `llvm-lib`/`llvm-ranlib`/`llvm-dlltool`): `CompareStringOrdinal`, Vista+, in the archiver `clang` invokes for static libs; and `raw_socket_stream.cpp`: [WSAPoll](https://learn.microsoft.com/en-us/windows/win32/api/winsock2/nf-winsock2-wsapoll), Vista+, in the socket listener's poll loop — reachable only in the shared `libLLVM` build, whose single library links socket code the standalone tools discarded). A static import of a function the OS lacks makes the whole `.exe` or `.dll` fail to start, so each is resolved at runtime via `GetProcAddress` (the same trick LLVM already uses for `PrefetchVirtualMemory`/`SetThreadInformation`). When absent: `realPathFromHandle` fails soft, `rename()` falls back to [MoveFileExW](https://learn.microsoft.com/en-us/windows/win32/api/winbase/nf-winbase-movefileexw) by path, `SHGetKnownFolderPath` falls back to `SHGetFolderPathW`+CSIDL, `remove_directories` falls back to the classic `SHFileOperationW`, the processor-group code collapses to a single `GetSystemInfo` group, `GetLargePageMinimum` returns 0 (large pages disabled), and `RegGetValueW` reports not-found so the (Vista-only) WER LocalDumps crash-dump lookups fall through to defaults, and `CompareStringOrdinal` falls back to an ASCII case-insensitive compare, and `WSAPoll` fails soft (the toolchain opens no sockets on those hosts). Separately, the psapi imports `EnumProcessModules`/`GetProcessMemoryInfo` are handled by building with `-DPSAPI_VERSION=1` (routes them to `psapi.dll` instead of the Win7+ `K32*` kernel32 forwarders), set in `llvm_win.sh`, not by this patch.

[clang-directorywatcher-pre-vista.patch](./llvm/clang-directorywatcher-pre-vista.patch) - A companion to `llvm-support-pre-vista.patch` for the shared-library build. `clang`'s `DirectoryWatcher` (compiled into `libclang-cpp.dll`) statically imports [GetFinalPathNameByHandleW](https://learn.microsoft.com/en-us/windows/win32/api/fileapi/nf-fileapi-getfinalpathnamebyhandlew) (Vista+); `lib/Support`'s own copy is already handled by that patch, but this `clang` call site is separate, and once all of `clang` is linked into one `libclang-cpp.dll`, its missing import would block the DLL (and therefore `clang.exe`) from loading on XP/2000. Resolves it via `GetProcAddress` and skips path canonicalization when absent (the watcher still runs via [ReadDirectoryChangesW](https://learn.microsoft.com/en-us/windows/win32/api/winbase/nf-winbase-readdirectorychangesw)). Self-guarding on `_WIN32_WINNT < 0x0600`, and only the Windows host compiles this file (a Linux host builds `DirectoryWatcher-linux.cpp`).

[llvm-tempfile-pre-vista.patch](./llvm/llvm-tempfile-pre-vista.patch) - Another companion to `llvm-support-pre-vista.patch`. LLVM's atomic file writes (`llvm::sys::fs::TempFile`, used by `FileOutputBuffer` and therefore by `llvm-rc`/`windres`, `lld`, `llvm-objcopy`, ...) write to a temp file then rename it onto the destination. On Windows `TempFile::keep()` renames through a HANDLE-based path that needs [SetFileInformationByHandle](https://learn.microsoft.com/en-us/windows/win32/api/winbase/nf-winbase-setfileinformationbyhandle) (Vista+), whose only fallback recovers the source path via [GetFinalPathNameByHandleW](https://learn.microsoft.com/en-us/windows/win32/api/fileapi/nf-fileapi-getfinalpathnamebyhandlew) (also Vista+) - both absent on XP/2000, so every `.res`/`.o`/`.exe` atomic write dies with `ERROR_CALL_NOT_IMPLEMENTED` ("This function is not supported on this system") right after the (working) preprocess/compile step - e.g. `windres foo.rc -O coff foo.res`. Pre-Vista it renames by path instead (`fs::rename` -> [MoveFileExW](https://learn.microsoft.com/en-us/windows/win32/api/winbase/nf-winbase-movefileexw), already provided by `llvm-support-pre-vista.patch`); the temp is opened `FILE_SHARE_DELETE` so it can be renamed while still open, and the `RemoveFileOnSignal` crash-cleanup handler `TempFile::create()` installs when the disposition call fails is kept intact. Gated with the support patch (`_WIN32_WINNT < 0x0601`); a no-op above XP.

[libunwind-psapi-dynamic.patch](./llvm/libunwind-psapi-dynamic.patch) - Resolves `libunwind`'s [EnumProcessModules](https://learn.microsoft.com/en-us/windows/win32/api/psapi/nf-psapi-enumprocessmodules) call dynamically (`LoadLibrary`/`GetProcAddress` on `psapi.dll`) so `libunwind` carries no *static* psapi import. That lets the LLVM build drop the `-Wl,--whole-archive -lpsapi` hammer it otherwise needed - which force-linked psapi's *entire* export surface into every tool, including Vista+ exports (`EnumProcessModulesEx`, `QueryWorkingSetEx`, `GetWsChangesEx`) that XP lacks and that then blocked the binaries from loading - for a plain `-lpsapi` at the end of the link line (pulls only referenced symbols). Paired with `-DPSAPI_VERSION=1` so the import targets `psapi.dll` (NT4+) rather than the Win7+ `K32*` kernel32 forwarders.

[llvm-win32-winnt-guard.patch](./llvm/llvm-win32-winnt-guard.patch) - Infrastructure for the LLVM Support pre-Vista fixes. LLVM's `Windows/WindowsSupport.h` `#undef`s and pins `_WIN32_WINNT` to `0x0601` (Win7) so the Windows SDK headers expose the Win7 API surface it wants - but that also pins the macro in every Support translation unit, defeating any compile-time `#if _WIN32_WINNT` guard in the pre-Vista patches. This uses `#pragma push_macro`/`pop_macro` so the Windows headers are still included at `0x0601` while the LLVM code body sees the caller's real legacy `_WIN32_WINNT`, plus the matching re-pins for the few Support files that pull in *extra* Windows headers past `WindowsSupport.h` (`Path.inc`'s `shlobj.h`, `raw_socket_stream.cpp`'s `winsock2.h`). Also lowers a stray Vista floor in `DebugInfo/PDB/DIA/DIASupport.h` to XP - dead code in a mingw build (LLVM's DIA/PDB reader needs the MSVC DIA SDK, never built here), but it preserves the "target XP if ever enabled" intent. Must be applied before `llvm-extra.patch`, which rides on the same `WindowsSupport.h`.

[llvm-extra.patch](./llvm/llvm-extra.patch) - Adds runtime Windows host-OS helpers to LLVM Support (`IsLegacyWindows`, `GetWindowsOSName`, `RunningWindows{2K,XP,Vista,7,10}OrGreater`, `IsSupportedWindows`, built on `RtlGetVersion`) and logs the detected host OS in `clang`/`lld`'s `--version` output, so it's obvious at a glance which legacy Windows a toolchain binary is actually running on. Windows-host only, and purely diagnostic/behavioral - it does **not** fix any load-time import (that is `llvm-support-pre-vista.patch` and friends).

[llvm-coff-fixes.patch](./llvm/llvm-coff-fixes.patch) - Bakes the output PE's default OS/subsystem version into `lld` at build time instead of stamping it through the toolchain wrappers. `lld` normally defaults to 6.0 (Vista), so binaries refuse to load on older Windows; this patches [lld/COFF/Config.h](https://github.com/llvm/llvm-project/blob/main/lld/COFF/Config.h) with a `#if _WIN32_WINNT` ladder so the default tracks the `_WIN32_WINNT` value `lld` is compiled with. The LLVM scripts pass `-D_WIN32_WINNT=$WIN32_WINNT` to the clang/lld build (scoped to the toolchain binaries, not the runtime libs), giving i586=4.0 / i686=5.0 / x86_64=5.2. It also makes `COFFModuleDefinition` parse `.def` version fields in any base (so hex `0x...` works). Applied ungated — the ladder is correct for any target.
